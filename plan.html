1

專案概述與目標

本專案旨在建立一個互動式產業知識地圖網站，以視覺化方式呈現特定產業的供應鏈關係、 關鍵公司及其財務數據。參考 semiEE（半導體供應鏈地圖）的架構， 但加入更深度的財務分析、未來預測、以及創新的互動體驗。

核心目標

  • 建立產業知識的互動式導航系統
  • 展示關鍵公司的營收、利潤、成長率、毛利率
  • 整合未來預測數據與趨勢分析
  • 視覺化呈現公司間的上下游供應鏈關係
  • 提供多維度的產業洞察與比較功能

目標用戶

  • 產業分析師與研究員
  • 投資人與基金經理
  • 供應鏈管理人員
  • 產業記者與顧問
  • 對產業生態有興趣的一般用戶

與 semiEE 的差異化

  • 從「靜態快照」進化為「動態時間軸」
  • 從「關係呈現」升級為「財務洞察」
  • 從「單一視角」擴展為「多維度圖層」
  • 增加深度的公司詳情側邊面板
  • 引入假設分析（What-If）模式

前端框架
React + Next.js
SSR/SSG 支援，效能優異，生態系成熟。支援靜態匯出，可先部署為純前端網站。
關係圖視覺化
AntV G6
螞蟻集團出品，專為關係網路設計。API 友善、中文文檔完善，處理大規模節點性能優秀。
財務圖表
ECharts
百度出品，圖表類型豐富。長條圖、折線圖、雷達圖、Sparkline 一應俱全。
樣式方案
Tailwind CSS
Utility-first CSS 框架，快速構建響應式介面，配合深色主題設計。
資料格式
JSON (初期) / API (後期)
初期用靜態 JSON 檔案，降低架構複雜度。後期可遷移到 REST API + 圖形資料庫。
部署平台
Vercel
與 Next.js 深度整合，免費額度充足，自動 CI/CD，支援自訂域名。
未來後端
Python + FastAPI
數據抓取（Scrapy）、清洗（Pandas）、API 服務一站式解決。
未來資料庫
Neo4j + PostgreSQL
Neo4j 儲存供應鏈圖譜關係，PostgreSQL + TimescaleDB 儲存時間序列財務數據。
策略建議：採用「靜態優先」策略 — 初期所有數據以 JSON 檔案形式管理， 配合 Next.js 的 Static Site Generation 部署為純前端網站。待數據量和更新頻率增長後， 再引入後端服務和資料庫。這樣可以最快速度上線，同時保留完整的擴展空間。

網站架構與頁面設計

首頁 / 產業總覽

  • 產業類別卡片網格（如：IC 設計、製造、封測、設備、材料）
  • 各類別的關鍵數據摘要（公司數量、總市值）
  • 全局搜尋列 — 快速搜尋公司名稱或代碼
  • 熱門公司快捷入口
  • 最新產業動態提要

互動式供應鏈地圖（核心頁面）

  • 可縮放的關係網路圖（AntV G6）
  • 節點大小 = 營收/市值（視覺權重）
  • 節點顏色 = 子領域分類
  • 連結粗細 = 供應關係強度
  • 懸浮 Tooltip：公司名、市值、近四季營收 Sparkline
  • 點擊展開側邊詳情面板
  • 篩選器：按國家、市值範圍、成長率

公司詳情側邊面板

  • Tab 1 — 公司概況：簡介、產品、官網
  • Tab 2 — 財務數據：五年營收/淨利/毛利率圖表
  • Tab 3 — 同業比較：雷達圖對比核心指標
  • Tab 4 — 上下游關係：供應商/客戶清單
  • Tab 5 — 未來展望：分析師預測、成長預估

公司排行榜 / 比較頁

  • 排行表格：依營收、成長率、毛利率排序
  • 多公司對比：選取 2-5 家公司並列比較
  • 氣泡圖：X 軸=成長率、Y 軸=毛利率、大小=營收
  • 匯出功能：CSV / 圖片下載

時間軸模式

  • 底部時間滑桿（可拖動年份）
  • 節點動態變化反映歷史演進
  • 播放/暫停按鈕，自動演進動畫
  • 關鍵事件標記（併購、IPO、重大政策）

What-If 模擬模式（進階）

  • 模擬斷鏈：關閉某節點，高亮受影響公司
  • 模擬併購：合併節點，計算新實體數據
  • 情境分析：匯率/關稅變動的影響推演

資料策略

財務數據來源

  • SEC EDGAR — 美股上市公司年報/季報（免費、權威）
  • 公開資訊觀測站 — 台股上市公司財報
  • Alpha Vantage / IEX Cloud — 經濟實惠的金融數據 API
  • Yahoo Finance — 基礎股價與財務摘要
  • Bloomberg / Refinitiv — 專業級（付費，後期考慮）

供應鏈關係來源

  • 公司年報 — 主要客戶/供應商揭露
  • 產業研究報告 — Gartner、IDC、TrendForce
  • 新聞與公告 — 合作協議、訂單公告
  • 專業資料庫 — Bloomberg Supply Chain
  • 手動研究 — 團隊深度產業調研

預測數據來源

  • 分析師共識預估 — 各券商研究報告彙整
  • 公司 Guidance — 法說會展望
  • 產業趨勢報告 — 產業研究機構預測
  • 自建模型 — 基於歷史趨勢的統計預測

更新策略

  • 財務數據 — 每季更新（跟隨財報季）
  • 供應鏈關係 — 每半年全面審查，重大事件即時更新
  • 預測數據 — 每季更新分析師共識
  • 數據版本控制 — 保留歷史快照，支援時間軸功能
  • 品質管控 — 所有數據標注來源與更新日期
初期策略：Phase 1 先以人工整理的 JSON 資料啟動，聚焦 20-30 家最關鍵的公司。 確保數據品質優先於數量。每筆資料附帶來源標註與更新日期，建立用戶信任。

Gemini 2.5 Pro 獨特見解

GEMINI INSIGHT

關鍵成功因素：從「呈現」到「洞察」

semiEE 告訴我們「誰是誰的供應商」，但沒有回答「所以呢？」。 本專案的成功關鍵在於回答這個問題。

  • 當用戶點擊台積電，不應只看到供應商是 ASML，而應能看到：台積電的營收成長與其資本支出的關聯性； ASML 營收有多大比例依賴台積電；若台積電毛利率下降，對上游供應商的議價壓力如何變化。
  • 數據的關聯性分析，才是核心價值。
GEMINI INSIGHT

創新互動：時間軸回放 + 假設分析

時間軸回放 — 增加時間滑桿（2015-2026），拖動時節點動態變化：

  • 節點大小代表市值，直觀看到公司興衰
  • 供應鏈關係遷移（貿易戰、新技術帶來的轉變）
  • 新進入者何時出現、如何挑戰巨頭

假設分析（What-If） — 提供沙盒模式：

  • 模擬併購：拖動公司 A 到公司 B，系統自動計算合併後的市佔率與營收
  • 模擬斷鏈：關閉 ASML 節點，紅色高亮所有受影響的下游晶片製造商
  • 這將工具從「回顧過去」進化為「推演未來」
GEMINI INSIGHT

多維度圖層疊加

  • 地理圖層 — 將節點疊加在世界地圖上，顯示全球供應鏈的地理分佈
  • 風險圖層 — 根據地緣政治、自然災害風險標示不同地區的風險等級
  • 技術專利圖層 — 顯示哪些公司在特定技術領域擁有核心專利
  • 這些跨領域的宏觀視角能幫助用戶發現單一維度下看不到的關聯和風險
GEMINI INSIGHT

技術選型建議

Gemini 強烈建議從 AntV G6 或 ECharts 開始，而非直接使用 D3.js：

  • G6/ECharts 能滿足 90% 的需求，且開發速度快數倍
  • D3.js 學習曲線極陡峭，開發週期長，適合追求極致客製化的後期階段
  • 資料庫建議混合使用：Neo4j（圖譜關係）+ PostgreSQL + TimescaleDB（時序財務數據）
  • 但初期可以用純 JSON 啟動，降低架構複雜度
GEMINI INSIGHT

領域專注策略

試圖涵蓋所有產業會導致深度不足。建議選擇一個最熟悉、最有數據資源的產業做深做透。 成為「某個領域的供應鏈權威」遠比成為「什麼都有但什麼都不精」的平台更有價值。 驗證產品後，再考慮橫向擴展。

展示層設計建議

視覺設計原則

  • 深色主題為主 — 專業感強，減少視覺疲勞
  • 色彩編碼一致性 — 同一子領域全站使用相同顏色
  • 資訊密度適中 — 避免過度擁擠
  • 漸進式揭露 — 由概覽到細節的分層展示
  • 動畫流暢 — 過渡動畫增強空間感

三層資訊架構

  • 第一層（全景）— 關係網路圖，節點大小代表市值
  • 第二層（懸浮）— Tooltip 展示核心指標 + Sparkline
  • 第三層（詳情）— 側邊面板完整公司分析
  • 用戶可在不離開主視圖的情況下，漸進式獲取資訊

互動體驗原則

  • 滑鼠懸停即時反饋 — Tooltip < 100ms 出現
  • 點擊反饋明確 — 選中節點高亮 + 關聯路徑淡化
  • 平滑的縮放與平移 — 支援觸控板手勢
  • 鍵盤導航 — Tab/方向鍵在節點間移動
  • 載入進度指示 — 大量節點時顯示載入狀態

潛在挑戰與解決方案

挑戰 風險描述 解決方案
數據取得與品質 財務數據來源分散、格式不一；部分非上市公司無公開數據 初期聚焦上市公司，使用標準化 JSON Schema；建立數據驗證流程；標注數據來源與日期
數據時效性 財務數據按季更新，供應鏈關係變動不定 建立季度更新排程；重大事件即時手動更新；所有數據帶時間戳
大量節點效能 數百個節點同時渲染可能導致頁面卡頓 AntV G6 內建虛擬化渲染；按類別分批載入；WebGL 渲染加速
供應鏈關係準確性 上下游關係常為非公開資訊，難以驗證 以公開年報揭露為主；標示「已確認」與「推測」關係；開放社群勘誤回報
預測數據可信度 預測本質上有不確定性，容易引起誤解 明確標示預測來源與假設條件；使用虛線/淡色區別預測與歷史數據；提供信賴區間
跨裝置體驗 複雜的關係網路圖在手機上難以操作 手機版提供簡化的清單視圖；桌面版才啟用完整互動地圖；觸控手勢優化
